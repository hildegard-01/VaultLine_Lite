# VaultLine Local — 하이브리드 전환 설계서 (최종)

> **중앙 경량 서버 + 로컬 SVN 저장소 하이브리드 아키텍처**  
> 기준 문서: VaultLine Local 설계명세서 v1.2 (로컬 기능) + VaultLine 서버 v3.2 (서버 기능)  
> 작성일: 2026-04-07  
> 목적: 기존 로컬 버전(거의 완성) 코드를 변경 없이 유지하면서, 서버 연동 계층을 위에 얹어 하이브리드로 전환  
> 이 문서 + v1.2 설계명세서 → 개발 착수 가능한 완전한 설계

---

## 1. 아키텍처 피벗 요약

### 1.1 왜 전환하는가

| 문제 | 설명 |
|------|------|
| 로컬 SVN GUI ≈ TortoiseSVN | 차별점이 부족 — SVN GUI는 이미 성숙한 도구가 있음 |
| 진짜 풀고 싶은 문제 | "서버 없이 팀원끼리 가볍게 문서 공유 + 버전관리" |
| P2P 공유의 한계 | 내 PC가 꺼지면 팀원 접근 불가 — 신뢰성 부족 |

### 1.2 새로운 아키텍처

```
┌─────────────────────────────────────────────────────┐
│              중앙 경량 서버 (아무 PC에서 실행 가능)       │
│  ┌────────────┐ ┌──────────┐ ┌──────────────────┐   │
│  │ Web Portal  │ │ User Mgmt│ │ SVN Mirror       │   │
│  │ (React)     │ │ JWT Auth │ │ (incremental     │   │
│  │ 미리보기/이력│ │ 공유/권한 │ │  dump → load)    │   │
│  └────────────┘ └──────────┘ └──────────────────┘   │
│  ┌──────────────────────────────────────────────┐   │
│  │ SQLite/PostgreSQL                              │   │
│  │ users, groups, shares, activity, approvals     │   │
│  └──────────────────────────────────────────────┘   │
│  No Docker. Express or FastAPI. 1~2GB RAM 충분.      │
└───────────────────────┬─────────────────────────────┘
                        │ API (HTTPS)
           ┌────────────┼────────────┐
           │            │            │
    ┌──────┴──────┐ ┌───┴─────┐ ┌───┴─────┐
    │ User A (PC)  │ │ User B  │ │ User C  │
    │              │ │         │ │         │
    │ SVN Repo     │ │ SVN Repo│ │ Browser │
    │ (file://)    │ │(file://)│ │ only    │
    │ 원본 데이터   │ │         │ │(웹 포탈) │
    │              │ │         │ │         │
    │ VaultLine App│ │VaultLine│ │No install│
    │ (Electron)   │ │ App     │ │ needed  │
    └──────────────┘ └─────────┘ └─────────┘
```

### 1.3 핵심 원칙

- **기존 로컬 버전 코드 변경 0** — 완성된 코드 위에 서버 연동 계층만 추가
- **서버 없이도 100% 작동** — 로그인 없이 로컬 SVN 기능 전부 사용 가능
- **UI는 VaultLine 서버 버전 풀 UI 복원** — 알림벨, 사용자 아바타, 공유 화면, 승인 화면, 관리자 화면 전부 복원
- **서버 기능은 커넥티드 모드에서만 활성화** — 오프라인 모드에서는 숨김/비활성

## 2. 이중 모드: 오프라인 / 커넥티드

### 2.1 모드 전환 흐름

```
앱 시작
  ├─ 서버 설정 없음 ────→ 오프라인 모드 (기존 코드 100%)
  │
  └─ 서버 설정 있음 ──→ 연결 시도
                         ├─ 성공 → 커넥티드 모드
                         └─ 실패 → 오프라인 모드 (자동 폴백, 주기적 재시도)
```

사용자가 언제든 수동 전환 가능: 설정 → 서버 → [연결] / [연결 해제] / [오프라인으로 작업]

### 2.2 기능별 모드 매트릭스

| 기능 | 오프라인 (로그인 없음) | 커넥티드 (로그인) |
|------|:-------------------:|:---------------:|
| 파일 탐색/커밋/Diff | ✅ 로컬 그대로 | ✅ 로컬 + 서버 미러 동기화 |
| 미리보기 (pdf.js) | ✅ 로컬 | ✅ 로컬 + 서버 미리보기(공유 파일) |
| 태그/즐겨찾기/휴지통 | ✅ 로컬 | ✅ 로컬 + 서버 동기화 |
| 검색 (FTS5) | ✅ 로컬 저장소만 | ✅ 로컬 + 팀 검색(서버 API) |
| 보호 잠금 (DB) | ✅ 로컬 | ✅ 로컬 (변경 없음) |
| SVN 잠금 (공유 저장소) | — | ✅ 서버 미러의 잠금 정보 표시 |
| 3방향 드래그앤드롭 | ✅ 로컬 | ✅ 로컬 (변경 없음) |
| 더블클릭 편집+수정감지 | ✅ 로컬 | ✅ 로컬 (변경 없음) |
| Pending Changes 바 | ✅ 로컬 | ✅ 로컬 (변경 없음) |
| 공유 링크 | 로컬 3방식 (zip/임시서버/클립보드) | 서버 영구 URL + 로컬 3방식 |
| 초대 링크 | docvault:// P2P | 서버 관리 https://server/invite/{token} |
| 사용자 관리 | 없음 | ✅ 서버 JWT auth, 역할 기반 |
| 알림 (벨 아이콘) | 숨김 | ✅ 서버 알림 표시 |
| 승인 워크플로우 | 숨김 | ✅ 서버 API 통해 활성화 |
| 관리자 화면 | 로컬 설정만 | ✅ VaultLine 풀 관리자 |
| 활동 로그 | 로컬만 | 로컬 + 서버 팀 로그 |
| 백업 | 로컬 hotcopy | 로컬 hotcopy + 서버는 메타데이터만 (파일 미러 없음) |

## 3. 서버 동기화 전략 — 메타데이터 전용 (파일 원본 저장 안 함)

### 3.1 설계 원칙: 서버에 파일을 넣지 않는다

기존 DocVault 서버 버전이 사용 불가능했던 이유가 서버 용량 문제였습니다.
SVN 저장소를 서버에 미러링하면 동일한 문제가 재발하므로, **서버에는 파일 원본을 저장하지 않습니다.**

| 전략 | 서버 저장량 | 10명×10GB | 판정 |
|------|-----------|----------|------|
| A. 풀 SVN 미러 | 100% | 100 GB | ❌ 기존 서버 버전과 동일한 문제 |
| B. HEAD 스냅샷만 | 5~10% | 5~10 GB | △ 여전히 큼 |
| C. 메타데이터만 | <1% | ~50 MB | ✅ 극경량 |
| **D. 메타데이터 + 선택적 미리보기 캐시** | **2~5%** | **~1 GB** | **✅ 채택** |

### 3.2 서버에 저장되는 것 / 저장되지 않는 것

| 데이터 | 서버 저장 | 크기 수준 | 비고 |
|--------|:-------:|----------|------|
| 커밋 로그 (리비전, 작성자, 메시지, 날짜, 변경 파일 목록) | ✅ | ~KB/커밋 | 매 커밋 시 앱이 JSON으로 push |
| 파일 트리 스냅샷 (현재 파일/폴더 구조, 크기, 리비전) | ✅ | ~KB/저장소 | 커밋 시 갱신 |
| 사용자, 그룹, 공유, 승인, 태그, 알림 | ✅ | ~KB | 서버 자체 DB |
| 미리보기 PDF 캐시 | ✅ (on-demand) | 제한 가능 | 누군가 열어볼 때만 생성+캐시. 자동 만료 |
| **SVN 저장소 원본 (파일 내용 + 이력)** | **❌** | — | **사용자 PC에만 존재** |
| **파일 다운로드용 사본** | **❌** | — | **소유자 PC에서 실시간 프록시** |
| **과거 리비전 파일 내용** | **❌** | — | **소유자 PC에서 on-demand 조회** |

### 3.3 커밋 시 앱 → 서버 push 프로토콜

매 커밋 후 앱이 서버에 보내는 데이터:

```json
POST /api/sync/commit
{
  "repo_id": "proj_a",
  "revision": 45,
  "author": "kimms",
  "message": "Q1 보고서 수치 수정",
  "date": "2026-04-07T10:30:00",
  "changed_files": [
    { "action": "M", "path": "/보고서/Q1_보고서.docx", "size": 263900 },
    { "action": "A", "path": "/보고서/첨부자료.pdf", "size": 125000 }
  ],
  "file_tree_snapshot": [
    { "path": "/보고서/Q1_보고서.docx", "size": 263900, "rev": 45 },
    { "path": "/보고서/첨부자료.pdf", "size": 125000, "rev": 45 },
    { "path": "/보고서/이전자료.xlsx", "size": 89400, "rev": 40 }
  ]
}
```

**페이로드 크기: 커밋당 1~5 KB** (메타데이터만, 파일 내용 없음)
비교: 풀 SVN dump는 같은 커밋에 200KB~50MB

### 3.4 웹 포탈에서의 파일 접근 (on-demand 프록시)

파일 내용이 필요한 순간에만 소유자 PC에서 실시간으로 가져옵니다:

```
웹 포탈에서 미리보기 클릭:
  1. 서버: preview_cache에 있는지 확인
  2. 캐시 HIT → 바로 서빙
  3. 캐시 MISS → 소유자 앱에 요청:
     서버 → WebSocket/SSE → 소유자 앱
     → 앱: svn cat repos/프로젝트A/보고서.docx | LibreOffice → PDF
     → 앱 → 서버로 PDF 전송
     → 서버: 캐시 저장 + 요청자에게 서빙
  4. 소유자 오프라인 → "파일 소유자가 오프라인입니다. 미리보기를 볼 수 없습니다."

웹 포탈에서 다운로드 클릭:
  1. 서버 → 소유자 앱에 파일 요청
  2. 앱 → 파일 스트리밍 → 서버 → 브라우저 (서버 디스크 미사용, 스트림 중계)
  3. 소유자 오프라인 → "파일 소유자가 오프라인입니다. 다운로드할 수 없습니다."
```

### 3.5 서버 용량 시뮬레이션

| 규모 | PC 총 용량 | 서버 메타데이터 | 미리보기 캐시 | **서버 총 사용량** |
|------|-----------|:-------------:|:-----------:|:----------------:|
| 5명 × 2GB | 10 GB | ~7 MB | ~200 MB | **~210 MB** |
| 10명 × 10GB | 100 GB | ~30 MB | ~1 GB | **~1 GB** |
| 15명 × 50GB | 750 GB | ~70 MB | ~3 GB | **~3 GB** |

미리보기 캐시에 상한(기본 5GB)을 설정하면 서버 용량이 영원히 제한됩니다.

## 3-A. 사용자 온라인/오프라인 상태 관리

메타데이터 전용 전략에서는 파일 내용을 소유자 PC에서 가져오므로, **누가 온라인인지**가 매우 중요합니다.

### 3-A.1 온라인 상태 감지

```
앱 → 서버:
  매 60초마다 heartbeat: POST /api/heartbeat { user_id, timestamp }
  앱 시작 시: POST /api/presence/online
  앱 종료 시: POST /api/presence/offline
  비정상 종료: heartbeat 3회 연속 실패 시 서버가 자동 offline 판정

서버 DB:
  users 테이블에 추가:
    last_heartbeat  DATETIME     -- 마지막 heartbeat 시각
    is_online       BOOLEAN      -- 현재 접속 여부
    last_seen       DATETIME     -- 마지막 접속 시각
```

### 3-A.2 상태 표시 위치

| 위치 | 표시 | 용도 |
|------|------|------|
| 관리자 대시보드 사용자 목록 | 🟢/🔴 도트 | 전체 사용자 접속 현황 |
| 사이드바 저장소 이름 옆 | 🟢/🔴 도트 | 해당 저장소 소유자의 접속 여부 → 미리보기/다운로드 가능 여부 |
| 공유받은 저장소 | "from 김민수 🟢" / "from 김민수 🔴 오프라인" | 게스트가 접근 가능 여부 판단 |
| 파일 미리보기/다운로드 버튼 | 소유자 오프라인이면 버튼 비활성 + 툴팁 | 실패 방지 |

### 3-A.3 오프라인 시 대처

| 상황 | 서버의 동작 |
|------|-----------|
| 웹 포탈 미리보기 요청 + 소유자 온라인 | 소유자 앱에 요청 → PDF 생성 → 캐시 → 서빙 |
| 웹 포탈 미리보기 요청 + 캐시 존재 | 캐시에서 바로 서빙 (소유자 상태 무관) |
| 웹 포탈 미리보기 요청 + 캐시 없음 + 소유자 오프라인 | "소유자가 오프라인입니다" 메시지 + [소유자 온라인 시 알림 받기] 버튼 |
| 다운로드 요청 + 소유자 오프라인 | "소유자가 오프라인이라 다운로드할 수 없습니다" |
| 파일 트리/커밋 이력/태그/검색 | 서버 DB에서 바로 서빙 (소유자 상태 무관) ✅ |
| 공유/승인/알림 | 서버 자체 기능 (소유자 상태 무관) ✅ |

## 3-B. 커밋 로그 보존 정책

커밋 로그가 무한히 쌓이면 서버 DB가 커집니다. 보존 정책을 두어 관리합니다.

### 3-B.1 보존 계층

| 계층 | 보존 기간 | 저장 위치 | 내용 |
|------|----------|----------|------|
| **핫 로그** | 최근 6개월 | 서버 DB (commit_log 테이블) | 전체 상세: 리비전, 작성자, 메시지, 변경 파일 목록 |
| **콜드 로그** | 6개월~2년 | 서버 DB (commit_log_archive) | 요약만: 리비전, 작성자, 메시지, 날짜. 변경 파일 목록 삭제 |
| **만료** | 2년 이상 | 삭제 (또는 CSV 내보내기 후 삭제) | 서버에서 제거. 원본은 사용자 PC SVN 이력에 영구 보존 |

### 3-B.2 관리자 설정 가능 항목

| 설정 | 기본값 | 옵션 | 설명 |
|------|--------|------|------|
| `log.hotRetentionMonths` | 6 | 3/6/12/24 | 핫 로그 보존 기간 |
| `log.coldRetentionMonths` | 24 | 12/24/36/unlimited | 콜드 로그 보존 기간 |
| `log.archiveAction` | `summarize` | summarize/delete/export | 핫→콜드 전환 시 동작 |
| `log.maxDbSizeMB` | 500 | 100~2000 | 커밋 로그 DB 최대 크기. 초과 시 강제 아카이브 |

### 3-B.3 자동 아카이브 스케줄

```
매일 03:00 서버 백그라운드 작업:
  1. 핫 로그 중 6개월 이상 → 콜드 로그로 이동 (변경 파일 목록 삭제)
  2. 콜드 로그 중 2년 이상 → 삭제 (또는 CSV 내보내기)
  3. commit_log 테이블 VACUUM
  4. 총 로그 DB 크기 확인 → 초과 시 경고 알림
```

**핵심**: 원본 SVN 이력은 사용자 PC에 영구 보존됩니다. 서버의 로그 삭제는 서버 웹 포탈에서의 조회 범위만 줄이는 것이지, 이력 자체가 소실되는 것이 아닙니다.

## 3-C. 서버 백업 제한 및 캐시 관리

### 3-C.1 미리보기 캐시 제한

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `cache.previewMaxSizeMB` | 5000 (5GB) | 미리보기 캐시 총 용량 상한 |
| `cache.previewMaxAgeDays` | 30 | 미접근 캐시 자동 삭제 기간 |
| `cache.previewCleanupTime` | 03:30 | 매일 자동 정리 시각 |

상한 초과 시: 가장 오래 접근하지 않은 캐시부터 순차 삭제.

### 3-C.2 서버 전체 용량 제한

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `server.maxStorageMB` | 10000 (10GB) | 서버 전체 사용 가능 용량 |
| `server.warningThreshold` | 80% | 경고 알림 발생 임계값 |
| `server.criticalThreshold` | 95% | 신규 캐시 생성 차단 임계값 |

```
용량 관리 흐름:
  80% 도달 → 관리자에게 경고 알림 ("서버 용량 80% 사용 중")
  95% 도달 → 미리보기 캐시 신규 생성 차단. 기존 캐시만 서빙
             → 관리자에게 긴급 알림 ("서버 용량 위험. 캐시 정리 필요")
  수동 정리: 관리자 → 시스템 → [캐시 정리] → 전체 또는 저장소별 선택 정리
```

### 3-C.3 서버 백업 범위 (경량)

서버 자체의 백업 대상은 극히 가볍습니다:

| 백업 대상 | 크기 수준 | 방법 |
|-----------|----------|------|
| 서버 DB (users, shares, commits, tags...) | ~50~200 MB | sqlite3 .backup 또는 pg_dump |
| 서버 설정 파일 | ~수 KB | 파일 복사 |
| 미리보기 캐시 | 1~5 GB | **백업 대상에서 제외** (재생성 가능) |

**미리보기 캐시는 백업하지 않습니다.** 소유자 PC에서 언제든 재생성 가능하므로, 복구 시 자연스럽게 다시 채워집니다. 이렇게 하면 서버 백업 크기가 **수백 MB 이내**로 유지됩니다.

### 3-C.4 사용자 PC 백업 (변경 없음)

사용자 PC의 SVN 저장소 원본 백업은 기존 로컬 버전 설계(svnadmin hotcopy + SQLite backup) 그대로 유지합니다. 서버가 이를 대체하거나 관리하지 않습니다. 각 사용자가 자기 앱에서 자기 백업을 관리합니다.

### 8.5 앱에서 서버 관리 화면 접근

관리자로 로그인 시 앱 사이드바 하단에 "관리" 메뉴가 표시됩니다. 클릭하면 사이드바가 관리 모드로 전환되고, 서버 API를 호출하여 관리자 화면을 렌더링합니다. 관리자 화면의 React 컴포넌트는 서버 API만 호출하므로, 앱(Electron)에서든 웹 브라우저에서든 동일한 코드로 동작합니다.

### 15.2 핵심 신규 테이블 상세

```sql
-- ═══ 저장소 레지스트리 (v3.2의 repositories 재설계) ═══
CREATE TABLE repos_registry (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    name            TEXT NOT NULL,
    description     TEXT DEFAULT '',
    owner_user_id   INTEGER NOT NULL REFERENCES users(id),
    type            TEXT DEFAULT 'personal',    -- personal / team
    group_id        INTEGER REFERENCES groups(id),  -- 팀 저장소인 경우
    latest_revision INTEGER DEFAULT 0,
    total_files     INTEGER DEFAULT 0,
    total_size_bytes BIGINT DEFAULT 0,         -- 소유자가 보고한 크기
    last_sync_at    DATETIME,
    status          TEXT DEFAULT 'active',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);
-- 주의: SVN 저장소 경로 없음. 파일 내용 없음. 메타데이터만.

-- ═══ 커밋 로그 (클라이언트에서 수신) ═══
CREATE TABLE commit_log (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    repo_id         INTEGER NOT NULL REFERENCES repos_registry(id),
    revision        INTEGER NOT NULL,
    author          TEXT NOT NULL,              -- 커밋한 사용자 ID
    message         TEXT DEFAULT '',
    committed_at    DATETIME NOT NULL,
    changed_files   TEXT,                       -- JSON: [{action, path, size}, ...]
    received_at     DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(repo_id, revision)
);

-- ═══ 파일 트리 스냅샷 (현재 HEAD 기준) ═══
CREATE TABLE file_tree (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    repo_id         INTEGER NOT NULL REFERENCES repos_registry(id),
    file_path       TEXT NOT NULL,
    is_directory    BOOLEAN DEFAULT 0,
    file_size       BIGINT DEFAULT 0,
    last_revision   INTEGER,
    last_author     TEXT,
    last_modified   DATETIME,
    UNIQUE(repo_id, file_path)
);

-- ═══ 커밋 로그 아카이브 (콜드 로그) ═══
CREATE TABLE commit_log_archive (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    repo_id         INTEGER NOT NULL REFERENCES repos_registry(id),
    revision        INTEGER NOT NULL,
    author          TEXT NOT NULL,
    message         TEXT DEFAULT '',
    committed_at    DATETIME NOT NULL,
    -- changed_files 제거 (요약만 보존)
    archived_at     DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(repo_id, revision)
);

-- ═══ 미리보기 캐시 메타데이터 ═══
CREATE TABLE preview_cache_meta (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    repo_id         INTEGER NOT NULL REFERENCES repos_registry(id),
    file_path       TEXT NOT NULL,
    revision        INTEGER NOT NULL,
    cache_file_path TEXT NOT NULL,              -- cache/preview/ 하위 경로
    file_size       INTEGER DEFAULT 0,
    last_accessed   DATETIME DEFAULT CURRENT_TIMESTAMP,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(repo_id, file_path, revision)
);

-- ═══ users 테이블 확장 (v3.2 + 온라인 상태) ═══
-- 기존 v3.2 users 컬럼 유지 + 아래 추가:
--   last_heartbeat  DATETIME     -- 마지막 heartbeat
--   is_online       BOOLEAN DEFAULT 0
--   last_seen       DATETIME     -- 마지막 접속 시각
```

## 10. 보안 아키텍처 (하이브리드)

### 13.1 보안 계층 — 로컬 앱 + 서버

하이브리드에서는 로컬 앱과 서버의 보안 모델이 다릅니다.

| 계층 | 로컬 앱 (오프라인) | 서버 (커넥티드) |
|------|:------------------:|:--------------:|
| 1. 네트워크 | — (로컬, 네트워크 없음) | HTTPS (Option C), 서버 방화벽 |
| 2. 인증 | — (로그인 없이 사용) | JWT + bcrypt (v3.2 동일) |
| 3. 인가 | — (단일 사용자) | 역할(admin/user) + 저장소별 권한 |
| 4. 데이터 | SQLite 파라미터 바인딩, 파일 검증 | ORM 파라미터 바인딩, 입력 검증 |
| 5. 감사 | 로컬 activity_log 19종 | 서버 activity_log (팀 전체) |

### 13.2 서버 인증 상세

VaultLine 서버 v3.2의 JWT 인증을 그대로 사용합니다:

- Access Token: 8시간, 앱 메모리 보관
- Refresh Token: 30일, HttpOnly/Secure/SameSite=Strict 쿠키
- 비밀번호: bcrypt(cost=12)
- 비밀번호 정책: 8자+, 대문자+숫자
- 초기 비밀번호 → 첫 로그인 시 변경 강제
- 로그인 실패: 5회 초과 시 15분 잠금

### 13.3 통신 보안

| 구간 | 보호 방식 |
|------|----------|
| 앱 ↔ 서버 (API) | HTTPS (Option C에서 Nginx TLS 적용). Option A/B에서는 HTTP (LAN 내부) |
| 앱 ↔ 서버 (WebSocket) | WSS (TLS 적용 시) 또는 WS (LAN 내부) |
| 커밋 메타 push | JWT 토큰 인증 + HTTPS |
| 파일 프록시 | JWT 토큰 인증 + WebSocket 채널 내 스트리밍 |
| 미리보기 push | JWT 토큰 인증 + WebSocket |

### 13.4 데이터 보호

| 보호 대상 | 방법 |
|-----------|------|
| 서버 DB (users, shares) | SQLAlchemy ORM 파라미터 바인딩 (SQL injection 방어) |
| 비밀번호 | bcrypt 해시 저장 (평문 미저장) |
| JWT 토큰 | 서명 검증 (HS256 또는 RS256) |
| 파일 프록시 | 요청자의 저장소 접근 권한 확인 후 중계 |
| 미리보기 캐시 | 접근 권한 확인 후 서빙 (공유 설정에 따라) |
| 앱 로컬 데이터 | OS 파일 권한 (사용자 계정 전용). SVN 저장소/DB 보호 |
| 로컬 백업 | AES-256 암호화 옵션 (v1.2 설계) |

### 13.5 배포 모드별 보안 수준

| 배포 | 네트워크 보안 | 적합 환경 |
|------|-------------|----------|
| Option A (베어 프로세스) | HTTP only. LAN 내부 신뢰 | 소규모 팀, 같은 사무실 |
| Option B (Docker) | HTTP. 포트 제한 가능 | 중규모, 사내 네트워크 |
| Option C (Docker+Nginx) | HTTPS (TLS), Rate Limit, HSTS | 보안 필요 환경, VPN 접속 |

### 13.6 하이브리드 특유의 보안 고려사항

| 위협 | 대응 |
|------|------|
| 서버 DB 유출 시 비밀번호 노출 | bcrypt 해시. 평문 미저장 |
| 파일 프록시를 통한 무단 파일 접근 | 서버가 저장소 권한 확인 후 프록시 |
| 소유자 PC 미리보기 생성 시 악의적 파일 | LibreOffice 격리 실행, 파일 확장자/크기 검증 |
| heartbeat 위조 (오프라인 사용자를 온라인으로) | JWT 토큰 필수. 토큰 없는 heartbeat 거부 |
| 커밋 메타데이터 위조 | JWT 토큰 + 서버 측 사용자-저장소 소유권 확인 |
| 미리보기 캐시 오염 | 캐시 키에 repo_id + file_path + revision 포함. 변조 시 불일치 감지 |

### 13.2 WebSocket 통신 프로토콜

```
1. HEARTBEAT (클라이언트 → 서버, 매 60초)
   { "type": "heartbeat", "user_id": "kimms", "timestamp": "..." }
   → 서버: last_heartbeat 갱신, is_online=true
   → 3회 연속 미수신: is_online=false, 관리자에게 경고

2. FILE_REQUEST (서버 → 클라이언트, on-demand)
   서버→클라이언트: { "type": "file_request", "req_id": "...",
                     "repo": "프로젝트A", "path": "/보고서.docx",
                     "action": "preview" | "download", "revision": "HEAD" }
   클라이언트→서버: { "type": "file_response", "req_id": "...",
                     "data": "<base64 PDF>" } 또는 스트리밍

3. NOTIFICATION (서버 → 클라이언트)
   서버→클라이언트: { "type": "notification",
                     "data": { "kind": "share", "message": "..." } }
   → 클라이언트: 벨 뱃지 업데이트 + 토스트

4. PREVIEW_PUSH (클라이언트 → 서버, 커밋 직후)
   클라이언트→서버: { "type": "preview_push", "repo": "...",
                     "path": "/보고서.docx", "revision": 45,
                     "data": "<base64 PDF>" }
   → 서버: cache/preview/ 에 저장 (eager cache)
```

### 13.3 미리보기 최적화: Eager Cache

50명 규모에서 on-demand 프록시만 사용하면 소유자 PC에 부하가 몰립니다. **커밋 시점에 미리 캐시를 채우는 전략**을 기본으로 합니다.

```
커밋 완료 후 앱의 동작:
  1. 메타데이터 JSON → POST /api/sync/commit (즉시, ~1KB)
  2. 변경된 파일의 미리보기 PDF 생성 (백그라운드)
     → LibreOffice 변환 또는 pdf.js 렌더링
  3. 생성 완료된 PDF → WS preview_push (파일당 수백 KB~수 MB)
  
결과:
  - 웹 포탈 접속 시 대부분 캐시 히트 → 소유자 PC 요청 불필요
  - 소유자 오프라인이어도 최근 커밋 파일의 미리보기는 항상 가능
  - 소유자 PC 부하: 커밋 직후 1회만 (이후 서버 캐시가 처리)
```

설정:
| 항목 | 기본값 | 설명 |
|------|--------|------|
| `preview.eagerPush` | `true` | 커밋 시 미리보기 자동 생성+push |
| `preview.eagerMaxSizeMB` | `50` | 이 크기 이하 파일만 자동 생성 (대용량 파일 제외) |
| `preview.eagerFormats` | `docx,xlsx,pptx,pdf,hwp` | 미리보기 대상 확장자 |

## 14. 서버 확장 전략 (2명 → 50명)

### 14.1 규모별 서버 사양

| 규모 | CPU | RAM | 디스크 | DB | 배포 방식 |
|------|-----|-----|--------|------|----------|
| 2~10명 | 아무 PC | 1 GB | 5 GB | SQLite | 단일 프로세스 (Option A) |
| 10~25명 | 2 core | 2 GB | 10 GB | SQLite → PostgreSQL | Docker 1컨테이너 (Option B) |
| 25~50명 | 4 core | 4 GB | 20 GB | PostgreSQL + Redis | Docker + Nginx (Option C) |

**비교**: 기존 v3.2가 50명 기준 4~8 core, 8~16GB RAM, **500GB 디스크**를 요구했습니다. 하이브리드는 **20GB 디스크 — 25배 절감**.

### 14.2 규모별 병목과 대응

| 규모 | 병목 | 증상 | 대응 |
|------|------|------|------|
| 10~25명 | SQLite 동시 쓰기 | 피크 시 "database is locked" | WAL 모드 확인 → 지속되면 PostgreSQL 전환 |
| 10~25명 | 미리보기 프록시 집중 | 같은 소유자에게 동시 요청 10+ | eager cache 활성화 + 소유자별 요청 큐 (최대 2개 동시) |
| 25~50명 | API 동시 접속 | 응답 지연 (200ms+) | `uvicorn --workers 4` + Nginx 리버스 프록시 |
| 25~50명 | WebSocket 관리 | 50개 영구 연결 관리 | Redis pub/sub으로 멀티워커 간 WS 상태 공유 |
| 40~50명 | 알림 팬아웃 | 공유 이벤트 → 30명에게 동시 알림 | 배치 알림 + 큐 기반 전송 |

### 14.3 DB 확장 경로

**코드 변경 없이 config만 변경하여 전환합니다. SQLAlchemy ORM이 양쪽 모두 지원.**

**Stage 1 — SQLite (2~20명):**
- 무설치. 파일 기반 DB
- WAL 모드로 동시 읽기 성능 확보
- 백업: 파일 복사 1줄

**Stage 2 — PostgreSQL (20~50명):**
- 진정한 동시 쓰기 지원
- 마이그레이션: `vaultline-server migrate --to postgresql`
- config.yaml의 DB URL만 변경

**Stage 3 — PostgreSQL + Redis (50명 근접):**
- Redis: 세션 캐시, WebSocket pub/sub, rate limiting
- 멀티워커 API 서버에서 상태 공유
- 필요할 때만 추가 (대부분 Stage 2로 충분)

### 14.4 미리보기 소유자별 요청 큐

50명 환경에서 특정 소유자에게 미리보기 요청이 몰리는 것을 방지합니다:

```
미리보기 요청 흐름:
  1. 웹 포탈 → 서버: GET /api/preview/{repo}/{path}
  2. 서버: 캐시 확인
     → HIT: 바로 서빙 (응답 <50ms)
     → MISS: 소유자 앱에 요청 필요
  3. 소유자별 요청 큐 확인:
     → 큐에 빈자리 있음 (max 2 동시): 즉시 요청
     → 큐 가득 참: 대기. "미리보기 생성 대기 중..." 표시
  4. 소유자 앱 → PDF 생성 → 서버 캐시 → 요청자에게 서빙
  5. 같은 파일을 다른 사람도 요청: 캐시에서 서빙 (중복 생성 방지)
```

## 15. 배포 옵션 (3가지)

### 15.1 Option A: 베어 프로세스 (가장 간단)

Docker 없이 직접 실행. 2~10명 소규모 팀.

```bash
# 설치
pip install vaultline-server

# 초기화 (최초 1회 — DB 생성, 관리자 계정 생성)
vaultline-server init
  → Admin ID: admin
  → Admin Password: ********
  → Port: 8080
  → Data directory: /data/vaultline

# 실행
vaultline-server start
  → http://0.0.0.0:8080 ready
```

장점: 3분 만에 설치 완료. 아무 여유 PC에서 실행 가능.
단점: 프로세스 관리를 수동으로 해야 함 (systemd 등록 권장).

### 15.2 Option B: Docker 단일 컨테이너 (권장)

5~25명. 안정적 운영.

```bash
docker run -d \
  --name vaultline-server \
  --restart unless-stopped \
  -p 8080:8080 \
  -v vaultline_data:/data \
  -e ADMIN_PASSWORD=yourpass \
  vaultline/server:latest
```

장점: 환경 일관성, 자동 재시작, 업데이트 용이.
단점: Docker 설치 필요.

### 15.3 Option C: Docker + Nginx (프로덕션)

25~50명. TLS + 리버스 프록시.

```yaml
# docker-compose.yml
version: '3.8'
services:
  server:
    image: vaultline/server:latest
    environment:
      - DATABASE_URL=postgresql://vaultline:pass@db:5432/vaultline
      - WORKERS=4
    volumes:
      - server_data:/data
    depends_on:
      - db

  db:
    image: postgres:16-alpine
    environment:
      - POSTGRES_DB=vaultline
      - POSTGRES_PASSWORD=pass
    volumes:
      - pg_data:/var/lib/postgresql/data

  nginx:
    image: nginx:alpine
    ports:
      - "443:443"
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
      - ./ssl:/etc/ssl/certs

volumes:
  server_data:
  pg_data:
```

컨테이너 3개: server(FastAPI) + db(PostgreSQL) + nginx.
기존 v3.2의 5컨테이너 대비 **SVN, LibreOffice, Worker 제거**.
